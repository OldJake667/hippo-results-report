## What MNIST Is

MNIST is a dataset of small black-and-white images of handwritten digits.
The digits are:

- `0`
- `1`
- `2`
- ...
- `9`

The model looks at each image and tries to predict which digit it is.

In this project, `mnist` is the default dataset in the config, so when the project is run normally, it tries to use MNIST first.

## MNIST Result

After fixing the environment and manually providing the MNIST files, the project was run successfully on MNIST.

The result was:

- training accuracy: `0.848`
- validation accuracy: `0.956`
- test accuracy: `0.961`

## What The MNIST Result Means

This means the model learned the task successfully.

The most important number is the **test accuracy**:

- `0.961` means about **96.1%** of the test images were classified correctly

In simple words:

- the model was able to recognize handwritten digits well
- the training pipeline worked
- the GPU setup also worked

So MNIST is the clearest example showing that the project can run and produce a meaningful result.

## What Adding Is

`Adding` is not downloaded from the internet.
It is an artificial dataset generated by the code itself.

The task is roughly this:

- the model sees a sequence of numbers
- it must learn which values matter
- then it must produce the correct sum

This dataset is used to test whether the model can handle long sequences and keep useful information.

## Adding Result

The `adding` dataset was run successfully.

The visible final result from the run was:

- loss: about `0.171`

## What The Adding Result Means

For this dataset, the main value is **loss**, not accuracy.

In simple words:

- lower loss is better
- a loss of about `0.171` shows the model was learning the task

So the `adding` run shows that the project also worked on a synthetic sequence dataset, not only on image data.

## What Copying Is

`Copying` is also an artificial dataset generated inside the project.

This task checks sequence memory.

In simple words, the model must:

- read a sequence
- remember important information from earlier in the sequence
- reproduce that information later

This is useful because it tests how well different model types handle memory over time.

## Copying Benchmark Setup

The `copying` dataset was used for the architecture comparison.

Four model families were compared:

- `recurrence` for recurrence
- `convolution` for convolution
- `attention` for attention
- `HiPPO` for state-space / HiPPO

Important note:

This repository does **not** contain a full S4 implementation.
So for the state-space part of the comparison, the closest built-in model in this project was `legs`, which is shown in the report as `HiPPO`.

## How The Benchmark Was Obtained

The benchmark was obtained by using the same training pipeline and changing only the model type.

The main training command pattern was:

```powershell
.\.venv-gpu\Scripts\python.exe train.py dataset=copying dataset.num_workers=0 train.epochs=2 model.cell=...
```

This means:

- the same dataset was kept: `copying`
- the same worker setting was kept: `dataset.num_workers=0`
- the same number of training epochs was kept: `train.epochs=2`
- only the model family was changed

This was done to make the comparison as fair as possible.

The concrete model choices were:

- `recurrence` for recurrence
- `convolution` for convolution
- `attention` for attention
- `HiPPO` for the built-in state-space / HiPPO model

In simple words, the idea was:

> Keep the task the same, keep the training conditions the same, and only change the kind of model.

That way, the runtime and accuracy differences are easier to compare.

## How The Synthetic Datasets Were Obtained

The synthetic datasets were not downloaded from the internet.
They were created automatically by the repository code itself.

For example:

- `adding` creates sequences of numbers and the model must learn the correct sum
- `copying` creates a sequence memory task where the model must remember an earlier pattern and reproduce it later

The main idea of `copying` is:

- first create a short random pattern
- then add a delay

So it is a memory test for sequence models.

To make benchmarking easier, a script was created:

This script automatically:

3. saves logs

Result files:

## Copying Benchmark Results

The following results were collected:

| Model | Family | Runtime (s) | Train Acc | Val Acc | Test Acc |
|---|---|---:|---:|---:|---:|
| `recurrence` | Recurrence | 485.03 | 0.109 | 0.009 | 0.008 |
| `convolution` | Convolution | 28.67 | 0.104 | 0.101 | 0.099 |
| `attention` | Attention | 44.63 | 0.126 | 0.123 | 0.125 |
| `HiPPO` | State-space / HiPPO | 639.89 | 0.094 | 0.132 | 0.130 |

## What These Benchmark Results Mean

These numbers compare two things:

- how long the model took to run
- how well the model performed on the `copying` task

### Runtime

Runtime means training time.

From the results:

- `convolution` was the fastest: `28.67` seconds
- `attention` was also fairly fast: `44.63` seconds
- `recurrence` was much slower: `485.03` seconds
- `HiPPO` was the slowest: `639.89` seconds

In simple words:

- convolution was the fastest model here
- attention was also efficient
- recurrence and HiPPO took much longer in this experiment

### Accuracy

Accuracy tells us how often the model gave the correct answer.

The most useful value here is **test accuracy**:

- `recurrence`: `0.008`
- `convolution`: `0.099`
- `attention`: `0.125`
- `HiPPO`: `0.130`

In simple words:

- `HiPPO` gave the best test result in this benchmark
- `attention` was close behind
- `convolution` was lower
- `recurrence` performed poorly in this short run

## Important Note About These Comparison Results

These benchmark results should be explained carefully.

They do **not** mean:

- convolution is always best
- recurrence is always bad
- attention is always best
- HiPPO is always best

Why not:

- all models were run only for `2` epochs
- this was a short comparison
- no deep tuning was done for each model

So the fair conclusion is:

> In this benchmark, the `convolution` model was the fastest, while the `HiPPO` model achieved the best test accuracy on the `copying` task.
